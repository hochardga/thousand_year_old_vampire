## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what has been built.
2. **Infrastructure before features.** The first phase creates a stable foundation so later feature work stays fast and coherent.
3. **Magic moment first.** The player must reach “I am already inside a vampire's life” as early as possible, not after several polish phases.
4. **Test as you go.** Every phase includes verification, and critical flows receive automated coverage before the roadmap moves on.
5. **Progressive enhancement.** Start with the simplest working version of ritual play, then layer in archive depth and polish.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent inspect it before starting the next phase.
7. **Tone is a feature.** UI decisions that damage the product's literary and solitary character count as product regressions, not cosmetic issues.
8. **Scope discipline is product quality.** If a task does not strengthen onboarding, prompt play, archive continuity, or launch readiness, it probably belongs later.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of the current phase.
2. **Read selectively:** Each phase lists its Reference sections. The agent should read only those sections from the PRD, vision doc, and GTM doc.
3. **Continue from the first unchecked task:** Tasks are intentionally ordered. Do not skip ahead unless a human explicitly changes the plan.
4. **Mark tasks complete immediately:** After finishing a task, update `- [ ]` to `- [x]` in this roadmap before moving on.
5. **Treat each phase as a review boundary:** When the phase is complete, create the phase branch, commit, push, and open a PR for automated review before starting the next phase.
6. **Verify before claiming completion:** Run the commands that prove the task works. Do not infer success from code changes alone.

### Session Tips

- **Don't load the entire docs set.** The reference sections exist to keep context narrow and accurate.
- **Don't widen scope mid-phase.** If a good idea is not required for the current goal, note it and move on.
- **Protect tone while coding.** Copy, layout, and state transitions are part of the product's core value, not optional polish.
- **Keep mobile honest.** Check important screens at narrow widths while building, not only at the end.
- **Review before moving on.** External review catches issues the coding agent will miss, especially around security, state, and edge cases.
